# Algorithm Practice 2026 (HPC/Quantum Preparation)

## 📌 プロジェクトの目的 (Purpose)
春休みの期間を活用し、コーディングテスト（Paiza Aランク相当）を突破するためのアルゴリズム力を構築する。
同時に、DevContainerとGitHubを用いた「摩擦ゼロの開発・Pushサイクル」を無意識レベルの習慣（Habit）に落とし込む。

## 📜 開発ルール・プロトコル (Habit Contract)

1. **15分ルール (Fail-Fast Rule)**
   - 問題に対して15分考えても解法（コード）が浮かばない場合、強制的に思考を停止する。
   - 躊躇なくAI（CIO）に壁打ちを依頼し、アルゴリズムのヒントを得てから実装に進む。時間を溶かすこと（タイムロス）を最大の悪とする。
2. **エントリーポイントの統一 (`main` rule)**
   - HPC研究室でのC/C++実装を見据え、Pythonであっても必ず `def main():` を定義し、`if __name__ == "__main__": main()` で呼び出す。
3. **変数名の認知コスト削減 (`tmp` rule)**
   - 競プロの学習用途に限り、一時的な変数は `tmp_sum`, `tmp_val` などを積極的に許容する。
   - 最終的な出力値は `ans` で統一し、問題のコンテキスト（りんご、気温など）に合わせた命名で脳のメモリを無駄遣いしない。
4. **Pythonicの追求**
   - まずは王道で堅牢な論理（ベースライン）を実装する。
   - その後、Python特有の機能（内包表記、ジェネレータなど）を用いた美しく短いコード（ワンライナー等）も学び、言語仕様への理解を深める。

## 🗺️ Aランク突破ロードマップ (Core Algorithms)
Paiza Bランク（多重ループの力技）からAランク（計算量の最適化）へ移行するための必須知識リスト。
$\mathcal{O}(N^2)$ の処理を、いかに $\mathcal{O}(N)$ や $\mathcal{O}(N \log N)$ に落とすかが鍵となる。

- [x] **1. 累積和 (Cumulative Sum)**
  - 区間の合計を $\mathcal{O}(1)$ で求める。配列の事前計算処理。
- [ ] **2. 二分探索 (Binary Search)**
  - ソート済み配列から特定の値を $\mathcal{O}(\log N)$ で見つける。「めぐる式二分探索」などの境界値探索も含む。
- [ ] **3. 幅優先探索 / 深さ優先探索 (BFS / DFS)**
  - グラフや迷路の最短経路、到達可能性を調べる。キュー(`deque`)や再帰関数の活用。Aランク最頻出。
- [ ] **4. 動的計画法 (DP: Dynamic Programming)**
  - 「ナップサック問題」に代表される、過去の計算結果を配列にメモして再利用する強力な手法。
- [ ] **5. ハッシュマップ / 集合 (Dict / Set)**
  - リストの `in` 検索（$\mathcal{O}(N)$）を、`set` を使って $\mathcal{O}(1)$ に落とすデータ構造の最適化。
- [ ] **6. しゃくとり法 (Two Pointers / Sliding Window)**
  - 条件を満たす連続する部分列を、2つのインデックスを伸縮させながら $\mathcal{O}(N)$ で見つける手法。
